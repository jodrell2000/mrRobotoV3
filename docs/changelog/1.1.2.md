# Version 1.1.2 - Configuration Token Handling Fix

**Release Date:** 12 May 2026  
**Type:** Bug Fix  
**Impact:** Critical - Fixes configuration token management and personality configuration merging

## Problem Statement

### Issue 1: Configuration Tokens Not Updating Behavior
The `!token add` command was creating custom token overlays for configuration tokens like `{timezone}`, `{locale}`, `{dateFormat}`, and `{timeFormat}`, but was NOT updating the underlying configuration values that built-in token functions actually use.

**Example Scenario:**
```
!token add {timezone} "America/Los_Angeles"
```
This would:
- ✅ Create a custom token that displays "America/Los_Angeles" when `{timezone}` is used in messages
- ❌ NOT update the actual `configuration.timezone` value used by `{currentTime}` token
- ❌ Result: `{timezone}` shows "America/Los_Angeles" but `{currentTime}` still uses "Europe/London"

This created a confusing situation where display tokens showed one value but the actual behavior used different configuration values.

### Issue 2: Personality Activation Overwrites Configuration
When activating a personality, the entire configuration object was replaced with the personality's saved configuration. If the personality was missing certain configuration values, those settings would be lost.

**Example Scenario:**
1. User manually sets `configuration.timezone` to "America/Los_Angeles" using direct data.json edit
2. User activates "The Janitor" personality which has `timezone: "Europe/London"` in its saved configuration
3. Result: Timezone reverts to "Europe/London", losing the manual change

This made it impossible to have partial personality configurations or maintain certain settings across personality switches.

## Solution

### Fix 1: Configuration Token Detection and Update
Modified `handleTokenCommand.js` to detect when a token being added is a configuration token and update both the configuration value AND create the custom token overlay.

**Configuration Tokens:**
- `{timezone}` → `configuration.timezone`
- `{locale}` → `configuration.locale`
- `{dateFormat}` → `configuration.dateFormat`
- `{timeFormat}` → `configuration.timeFormat`

**New Behavior:**
```javascript
// User command
!token add {timezone} "America/Los_Angeles"

// Bot now does BOTH:
1. Updates configuration.timezone = "America/Los_Angeles"
2. Creates custom token {timezone} = "America/Los_Angeles"

// Result:
- {currentTime} uses "America/Los_Angeles" timezone ✅
- {timezone} displays "America/Los_Angeles" ✅
- Bot response confirms: "Configuration updated: {timezone} = "America/Los_Angeles" (affects timezone behavior)"
```

### Fix 2: Configuration Merging on Personality Activation
Modified `handlePersonalityCommand.js` to merge personality configuration with existing configuration instead of replacing it entirely.

**New Behavior:**
```javascript
// Before (v1.1.1 and earlier)
await dataService.setValue('configuration', otherConfig); // Overwrites entire config

// After (v1.1.2)
const currentConfig = dataService.getValue('configuration') || {};
const mergedConfig = { ...currentConfig, ...otherConfig };
await dataService.setValue('configuration', mergedConfig); // Merges configs
```

This ensures:
- Values in the personality configuration take precedence
- Values not present in the personality configuration are preserved from current settings
- No configuration values are lost during personality switches

## Technical Changes

### Modified Files
1. **src/commands/Edit Commands/handleTokenCommand.js**
   - Added `CONFIGURATION_TOKENS` mapping object
   - Updated `handleAddToken()` to detect configuration tokens
   - Added configuration update logic before custom token creation
   - Enhanced response message to indicate configuration updates

2. **src/commands/Edit Commands/handlePersonalityCommand.js**
   - Updated `handleActivatePersonality()` to merge configurations
   - Added configuration retrieval before setting merged values
   - Preserves missing configuration values from current settings

### Code Additions

**handleTokenCommand.js:**
```javascript
const CONFIGURATION_TOKENS = {
    '{timezone}': 'timezone',
    '{locale}': 'locale',
    '{dateFormat}': 'dateFormat',
    '{timeFormat}': 'timeFormat'
};

// In handleAddToken():
const normalizedName = tokenName.startsWith('{') ? tokenName : `{${tokenName}}`;
const configKey = CONFIGURATION_TOKENS[normalizedName];
if (configKey) {
    await dataService.loadData();
    await dataService.setValue(`configuration.${configKey}`, tokenValue);
    // ... then create custom token overlay
}
```

**handlePersonalityCommand.js:**
```javascript
// In handleActivatePersonality():
const currentConfig = dataService.getValue('configuration') || {};
const mergedConfig = { ...currentConfig, ...otherConfig };
await dataService.setValue('configuration', mergedConfig);
```

## User-Facing Changes

### Command Behavior
**`!token add {timezone} <value>`** (and other configuration tokens):
- Now updates the actual configuration used by built-in tokens
- Provides confirmation message: `"Configuration updated: {timezone} = "America/Los_Angeles" (affects timezone behavior)"`
- Built-in tokens like `{currentTime}` immediately respect the new value

### Personality System
**`!personality activate <name>`**:
- Now preserves configuration values not present in the personality
- Prevents losing manually configured settings during personality switches
- Example: If you've set a specific timezone, it won't be lost when activating a personality that doesn't specify a timezone

### Post-Upgrade Steps

   ```
   !token add {timezone} "America/Los_Angeles"
   ```

2. **Update Active Personality** (recommended):
   ```
   !personality update
   ```
   This captures the corrected configuration in your current personality's snapshot.

   ```
   !token test {currentTime}
   !token test {timezone}
   ```
