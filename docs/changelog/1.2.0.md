# Release v1.2.0 - Dynamic Web Documentation

**Release Date:** TBD  
**Type:** Minor Release (New Feature)

## Overview

This release adds a web-based documentation server that provides access to bot commands, status information, and configuration through a web browser. The server runs on port 8080 and is accessible via HTTP.

**⚠️ IMPORTANT:** This feature requires Oracle Cloud Infrastructure configuration (see below).

## ⚠️ REQUIRED: Oracle Cloud Infrastructure Setup

### MUST Configure Ingress Rule for Port 8080

**This configuration is mandatory for the web documentation feature to be accessible.** Without this ingress rule, port 8080 will be blocked by Oracle Cloud's firewall and the documentation server will not be reachable from outside the VM.

#### Step-by-Step Instructions

1. **Log in to Oracle Cloud Console**
   - Go to: https://cloud.oracle.com
   - Sign in with your account

2. **Navigate to Your Instance**
   - Click the hamburger menu (☰) → **Compute** → **Instances**
   - Click on your VM instance name

3. **Find the Subnet**
   - Scroll to the **Primary VNIC** section
   - Click on the **Subnet** name (e.g., "Public Subnet-vcn-...")

4. **Open Security Lists**
   - In the left sidebar, click **Security Lists**
   - Click on the security list name (e.g., "Default Security List for vcn-...")

5. **Add Ingress Rule**
   - Click the **"Add Ingress Rules"** button
   - Fill in the following:
     - **Stateless:** Leave **unchecked** (No)
     - **Source Type:** CIDR
     - **Source CIDR:** `0.0.0.0/0` (allows all IPs - see security note below)
     - **IP Protocol:** TCP
     - **Source Port Range:** (leave empty)
     - **Destination Port Range:** `8080`
     - **Description:** `Web documentation server`
   - Click **"Add Ingress Rules"** at the bottom

6. **Verify the Rule**
   - Wait 10-30 seconds for the rule to propagate
   - Test from your local machine:
     ```bash
     curl http://YOUR-VM-IP:8080/health
     # Should return: ok
     ```
   - Or open in browser: `http://YOUR-VM-IP:8080/health`

#### Security Considerations

**Source CIDR Options:**
- `0.0.0.0/0` - Allows access from any IP address (public documentation)
- `YOUR-IP/32` - Restricts access to only your IP address
- `YOUR-NETWORK/24` - Restricts access to your network range

**Recommendation:** Start with `0.0.0.0/0` for public documentation. The web server only serves read-only documentation with proper security controls (see security section below).

## Problem Statement

Users needed an easy way to:
- View available chat commands and their usage
- Check bot status and version information
- Access command documentation with images and examples
- Monitor bot health and connectivity

Previously, all documentation was in markdown files in the repository, requiring users to have access to the codebase or GitHub.

## Solution

Implemented a web server that serves dynamic documentation pages:
- Health check endpoint for monitoring (`/health`)
- Status page with version, uptime, and bot state (planned)
- Command documentation with examples (planned)
- Chat commands with images (planned)

## Security

### Built-in Protection

The HTTP server implements strict path whitelisting to prevent unauthorized access:

**Protected Resources:**
- `.env` file (contains API keys and tokens) → **BLOCKED**
- `data.json` and all data files → **BLOCKED**
- `package.json` and source code → **BLOCKED**
- All file system paths → **BLOCKED**

**How It Works:**
- Server validates every incoming request URL
- Only explicitly whitelisted paths are served
- Currently whitelisted: `/health` (returns "ok")
- All other paths return 404 Not Found
- Prevents directory traversal attacks (e.g., `/../.env`)

**Verification:**
```bash
# Allowed endpoint
curl http://YOUR-IP:8080/health
# Returns: ok (200 OK)

# Blocked sensitive files
curl http://YOUR-IP:8080/.env
# Returns: Not Found (404)

curl http://YOUR-IP:8080/data.json
# Returns: Not Found (404)
```

### Why Public Access is Safe

The web server is **read-only** and serves only approved documentation endpoints. Even with `0.0.0.0/0` (public access), the server:
- Cannot modify bot configuration
- Cannot access sensitive files
- Cannot execute commands
- Only serves whitelisted documentation pages

All bot control remains exclusively through authenticated chat commands.

## Deployment Instructions

### Standard Deployment

```bash
# Deploy to Oracle Cloud with latest image
IMAGE_TAG=1.2.0 ORACLE_IP=YOUR-IP ./scripts/deploy-to-oracle.sh
```

### What Happens During Deployment

1. Uploads .env file to VM
2. **Automatically injects `WEB_DOCS_URL=http://YOUR-IP:8080`** into .env
3. Pulls Docker image with tag 1.2.0
4. Starts container with port 8080 mapped
5. Bot is accessible at `http://YOUR-IP:8080`

### Verify Deployment

```bash
# Check health endpoint
curl http://YOUR-IP:8080/health
# Should return: ok

# Check VERSION file
ssh ubuntu@YOUR-IP 'docker exec mrroboto cat VERSION'
# Should show JSON with version info

# Check WEB_DOCS_URL was configured
ssh ubuntu@YOUR-IP 'grep WEB_DOCS_URL ~/mrroboto/.env'
# Should show: WEB_DOCS_URL=http://YOUR-IP:8080

# Check container health
ssh ubuntu@YOUR-IP 'docker ps'
# Status should show "healthy"
```

## Upgrade Notes

### From v1.1.x to v1.2.0

1. **Deploy the new version** using standard deployment process
2. **Configure Oracle Cloud firewall** following instructions above
3. **Verify health endpoint** is accessible
4. No data migration or configuration changes required (WEB_DOCS_URL is auto-configured)

### Rollback Procedure

If you need to rollback to v1.1.3:

```bash
# Deploy previous version
IMAGE_TAG=1.1.3 ORACLE_IP=YOUR-IP ./scripts/deploy-to-oracle.sh

# Remove ingress rule from Oracle Cloud (optional)
# The rule won't cause issues, but port 8080 won't be used
```
