# Version 1.2.2 - ML Service 429 Error Handling

**Release Date:** May 31, 2026  
**Type:** Patch Release

## Overview

Optimized machine learning service error handling to detect and respond appropriately to Google AI API quota/rate limit errors (HTTP 429). When quota is exhausted, the service now immediately stops trying additional models instead of wasting API calls on models that will also fail.

## 🔧 Fixed

### Machine Learning Service (`machineLearningService.js`)

- **Optimized:** Added specific detection for 429 rate limit/quota exceeded errors
- **Fixed:** Stop model fallback chain immediately when API quota is exhausted
- **Improved:** Enhanced error logging with specific message about resource exhaustion
- **Fixed:** Prevent unnecessary API calls to backup models when quota limit is reached

## Impact

### Before This Fix
When API quota was exhausted:
1. Primary model (gemma-4-31b-it) → 429 error
2. Fallback model (gemma-4-26b-a4b-it) → 429 error (wasted API call)
3. Any additional models → 429 errors (more wasted calls)
4. Generic error logged

### After This Fix
When API quota is exhausted:
1. Primary model (gemma-4-31b-it) → 429 error detected
2. Specific log message: "API quota/rate limit exceeded (429). Resources have been exhausted for this API key. No further models will be attempted."
3. Immediately returns error message to user
4. No wasted API calls to backup models

### Error Detection
The service now detects 429 errors through multiple methods:
- HTTP status code 429
- Error code 429
- Error message containing "429", "quota", or "rate limit"

### Non-429 Errors
Other errors (network issues, model failures, etc.) continue to use the full fallback chain as before.

## Technical Details

### Implementation
Added conditional logic in the model fallback loop to:
1. Check if caught error is a 429/quota error
2. Log specific error message about resource exhaustion
3. Break out of fallback loop immediately
4. Return generic error message to user

### Logging
- **429 errors:** `logger.error()` with detailed quota exhaustion message
- **Other errors:** `logger.warn()` with model name and error message (existing behavior)

## Migration Notes

No migration required. This is a transparent optimization that:
- Reduces API calls when quota is exhausted
- Provides clearer error logs for debugging
- Maintains existing user-facing error messages
