### Automated Deployment Scripts

Comprehensive automation for deploying and managing the bot on Oracle Cloud Infrastructure.

#### Deploy Script Features

- **SSH Authentication Flexibility**
  - Supports SSH agents (1Password, Secretive, ssh-agent)
  - Optional SSH key file support via `ORACLE_SSH_KEY` environment variable
  - Works with default SSH keys (~/.ssh/id_rsa, etc.)

- **Docker Image Version Control**
  - Configurable via `IMAGE_TAG` environment variable
  - Defaults to `latest` tag if not specified
  - Supports any tag available in GitHub Container Registry
  - Example: `IMAGE_TAG=1.0.0 ORACLE_IP=xxx ./scripts/deploy-to-oracle.sh`
  
- **Docker Environment Handling**
  - Automatically strips quotes from `.env` values (Docker vs docker-compose compatibility)
  - Removes `GCS_BUCKET_NAME` automatically (not needed on OCI)
  - Validates environment file before upload

- **Deployment Process**
  - Tests SSH connection before starting
  - Creates remote directory structure
  - Uploads `.env` file with transformations
  - Optionally uploads data directory (`--upload-data`)
  - Pulls latest Docker image from GitHub Container Registry
  - Stops old container gracefully (30-second timeout)
  - Starts new container with correct configuration
  - Optionally shows logs (`--logs`)

#### Sync Script Features

- **Bidirectional Data Transfer**
  - Download mode (default): Pull VM data to local
  - Upload mode (`--upload`): Push local data to VM  
  - Backup mode (`--backup`): Create timestamped backup before download

- **Smart Filtering**
  - Excludes `*_example` files from uploads
  - Preserves file permissions and timestamps
  - Shows file statistics after sync

#### Data Persistence on OCI

Unlike Google Cloud Run's stateless containers, Oracle Cloud VMs provide:

- **Persistent Disk Storage**
  - Data stored directly on VM boot volume
  - No external storage service needed
  - Survives container restarts and updates
  - Part of Always Free tier (200GB included)

- **Simple Backup Strategy**
  - Download data periodically: `sync-oracle-data.sh --backup`
  - Upload changes before deployment: `deploy-to-oracle.sh --upload-data`
  - VM disk provides primary persistence

### Message Handling Improvements

Enhanced handling of corrupted or invalid messages from CometChat.

#### Changes

- **Silent Filtering of Unknown Users**
  - Messages with empty/invalid sender UUIDs are silently skipped
  - Prevents log spam from corrupted CometChat messages
  - No warnings generated for duplicate corrupted messages

- **Improved User Validation**
  - Validates UUID before fetching private messages
  - Skips users with invalid/empty UUIDs during message polling
  - Prevents unnecessary API calls for corrupted user entries

- **Cleaner Logs**
  - Removed repetitive duplicate message warnings
  - Silent handling of edge cases
  - Focus on actionable error messages only

#### Technical Details

- Early exit in `_processSingleMessage()` for messages without valid senders
- UUID validation in `_fetchNewPrivateMessages()` user loop
- Removed warning logs from `_updateMessageTracking()` for empty senders

### Gemma 4 Model Support

Switched from Gemma 3 models to newer Gemma 4 models with enhanced reasoning capabilities.

#### Changes

- **Primary Model:** `gemma-3-27b-it` → `gemma-4-31b-it`
- **Secondary Model:** `gemma-3-12b-it` → `gemma-4-26b-a4b-it`
- **Model Discovery:** Auto-discovery now filters to `gemma-4` prefix only
- **Thinking Config:** Added `thinkingConfig` with `includeThoughts: true` and `thinkingLevel: "HIGH"` for improved reasoning

#### Benefits

- Enhanced reasoning capabilities via Gemma 4's thinking features
- More consistent responses with better context understanding
- Automatic fallback chain ensures availability

### ML Configuration Improvements

Refined Google GenAI API parameters to reduce response repetition and improve quality.

#### Changes

- **Temperature:** `2.0` → `0.8` (reduced from overly random value)
- **Top-P:** Added `0.8` (nucleus sampling for diversity)
- **Top-K:** Added `20` (token sampling limit)
- **Max Output Tokens:** Added `1024` (explicit response length limit)
- **Thinking Config:** Added for Gemma 4 reasoning capabilities

#### API Parameter Investigation

- Confirmed Google GenAI API does not support `presencePenalty` or `frequencyPenalty` (OpenAI-specific)
- Identified supported parameters: `temperature`, `topP`, `topK`, `maxOutputTokens`, `stopSequences`
- Updated configuration to use only supported parameters
- Fixed inline comments to accurately reflect actual values

#### Testing

- All 41 ML service tests passing
- Updated test expectations to match new configuration
- Validates model selection, configuration parameters, and thinking config

## 🚀 Deployment Notes

This release enables production deployment to Oracle Cloud Infrastructure with:

1. **Zero-cost hosting** via Always Free tier (£0/month forever)
2. **Persistent data** on VM disk (no external storage needed)
3. **24/7 availability** with no time limits or cold starts
4. **Simple deployment** via automated scripts
5. **SSH agent support** for modern authentication workflows

### New Installation on Oracle Cloud

Follow the complete guide in [CLOUD_HOSTING.md](../CLOUD_HOSTING.md):

1. Create Oracle Cloud account (identity verification required, no charges)
2. Create VCN via wizard (networking infrastructure)
3. Create VM.Standard.E2.1.Micro instance (Always Free eligible)
4. Connect via SSH (Windows/macOS/Linux instructions provided)
5. Install Docker on VM
6. Deploy bot: `ORACLE_IP=xxx.xxx.xxx.xxx ./scripts/deploy-to-oracle.sh --upload-data --logs`
7. Monitor: `ssh ubuntu@xxx.xxx.xxx.xxx 'docker logs -f mrroboto'`

### Migration from Local Docker

If you're currently running locally:

1. Follow Oracle Cloud setup in CLOUD_HOSTING.md (Steps 1-6)
2. Deploy with data upload: `ORACLE_IP=xxx.xxx.xxx.xxx ./scripts/deploy-to-oracle.sh --upload-data --logs`
3. Verify bot joins hangout
4. Stop local Docker instance
5. Download VM data periodically: `ORACLE_IP=xxx.xxx.xxx.xxx ./scripts/sync-oracle-data.sh --backup`

## ⚠️ Breaking Changes

**Environment Variable Changes:**

- **Removed:** `GCS_BUCKET_NAME` - No longer needed for Oracle Cloud deployment
  - If you were using Google Cloud Storage, data is now stored on VM disk
  - For Oracle Cloud: Data persists on VM disk automatically
  - For local Docker: No change to behavior

**Migration Impact:** 

If upgrading from a version that used Google Cloud Storage:
- Your GCS bucket data will remain in Google Cloud
- Download data from GCS before migrating: `bash scripts/sync-cloud-data.sh --download`
- Upload to Oracle VM: `ORACLE_IP=xxx.xxx.xxx.xxx ./scripts/sync-oracle-data.sh --upload`
- Optionally delete GCS bucket if no longer needed
