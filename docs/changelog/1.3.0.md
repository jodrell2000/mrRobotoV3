# Version 1.3.0 - OpenChat Migration & Private Message Optimization

**Release Date:** 2026-06-16  
**Type:** Minor Release

## Overview

Migrated the bot's chat service from CometChat to OpenChat (tt.fm's official CometChat-compatible chat endpoint). OpenChat provides 100% API compatibility with CometChat, requiring only a base URL change. This migration improves service integration with the tt.fm platform while maintaining all existing chat functionality. Additionally, removed Google Cloud dependencies to streamline infrastructure (Oracle Cloud deployment remains unchanged).

Also implemented significant performance improvements for private message fetching by refactoring from sequential to parallel execution with per-user dynamic backoff, reducing fetch time from 6.5s+ to 1-2s while fixing timeout detection and backoff escalation bugs.

## 🚀 Private Message Backoff Improvements

### Architecture Refactoring

Refactored private message fetching from sequential to parallel execution with per-user dynamic backoff:

**Before:**
- Sequential fetching of users (one after another)
- 500ms delay between users
- 7 users × 500ms = 3.5s+ per cycle
- Global 5-second timeout (often exceeded)
- All users failing with consistent 900ms timeout

**After:**
- Parallel fetching of all users simultaneously
- Individual per-user timeout and backoff
- Fetch completes in 1-2s (all users in parallel)
- Each user escalates interval independently: 1000ms → 2000ms → 3000ms... → 10000ms max
- Interval resets to 1000ms on successful fetch

### Bug Fixes

**Fixed: Case-Sensitivity Timeout Detection**
- Error message: `"Fetch timeout after 900ms"` (lowercase 't')
- Detection code: checking for `'Timeout'` (uppercase 'T')
- Result: Timeouts never recognized, backoff never triggered
- Fix: Use consistent marker `PRIVATE_MESSAGE_TIMEOUT` for guaranteed matching

**Fixed: Spurious Timeout Warnings**
- `setTimeout` was still firing after fetch completed successfully
- Promise.race resolves with first winner, but setTimeout keeps running
- At ~900ms after start, the late timeout would fire and log warning
- Fix: Store `timeoutId` and `clearTimeout()` when fetch completes first

**Fixed: Overall Timeout Preventing Backoff**
- 15-second batch timeout was cutting off all parallel fetches
- If one user backed off to 10 seconds, batch timeout would kill fetch at 15 seconds
- Prevented per-user backoff from ever reaching full escalation
- Fix: Removed batch timeout entirely; rely on per-user timeouts via Promise.allSettled()

### Implementation Details

**Modified Files:**
- `src/lib/bot.js`:
  - Added `this.privateMessageUserIntervals = {}` - per-user backoff state
  - Refactored `_fetchNewPrivateMessages()` - parallel Promise.allSettled() orchestration
  - Added `_fetchPrivateMessagesForUser()` - individual user handler with timeout/backoff
  - Updated `processNewPrivateMessages()` - removed overall timeout

**Per-User State Structure:**
```javascript
privateMessageUserIntervals[userUUID] = {
  interval: 1000,      // Current retry interval (1s base)
  backoffStep: 1000,   // Increment per timeout
  maxInterval: 10000   // Maximum 10 seconds
}
```

**Timeout Calculation:**
- Timeout = `Math.floor(interval × 0.9)` (90% of current interval)
- At 1000ms interval → 900ms timeout
- At 2000ms interval → 1800ms timeout
- At 10000ms interval → 9000ms timeout

### Performance Impact

**Positive:**
- Overall fetch time reduced from 6.5s+ to 1-2s
- Parallel execution eliminates sequential delays
- Per-user backoff handles variable API performance gracefully

**Neutral:**
- No impact on resource consumption
- Similar CPU and memory usage
- Better utilization of parallel execution
